# Конституция проекта — bsl-atlas (personal fork)

> Этот документ является главным управляющим документом проекта.
> Агент обязан прочитать его перед любым действием.
> Все последующие решения — спеки, планы, задачи — должны соответствовать
> принципам, изложенным здесь. При противоречии конституция имеет приоритет.

---

## 1. Назначение проекта

Персональный форк bsl-atlas, открытый для 1С-сообщества.

**Основные цели (в порядке приоритета):**
1. Анализ зависимостей между модулями BSL в 1С конфигурациях
2. Генерация документации по 1С конфигурации
3. Интеграция с AI-агентами (Roo Code, Qwen Code) через MCP
4. Наполнение knowledge graph (Graphiti, Neo4j) сущностями из BSL

**Проект поддерживается единолично.** Решения о приоритетах, архитектуре
и trade-off'ах принимает владелец репозитория, не агент.

---

## 2. Brownfield: работа с существующим кодом

Это форк существующего репозитория. Это меняет всё.

**Агент обязан:**
- Изучить существующие паттерны в области изменяемого кода
  ДО написания любого нового кода
- Следовать конвенциям оригинального репо в зоне изменений
- Добавлять новый код, не переписывая существующий

**Агент не имеет права:**
- Изменять, рефакторить или удалять существующий код оригинального репо
- Исключение: явное указание в спеке с обоснованием

---

## 3. Разработка через тестирование (TDD)

**Правило абсолютное, без исключений:**

```
1. Написать падающий тест (red)
2. Убедиться что тест падает по правильной причине
3. Написать минимальный код для прохождения теста (green)
4. Рефакторить не меняя поведения (refactor)
```

Агент не может перейти к шагу 3 не выполнив шаги 1–2.
Если агент не может написать тест первым — он обязан остановиться
и объяснить причину, а не пропустить шаг.

## 4. Стандарты качества кода

### Обязательные проверки перед каждым коммитом

| Инструмент | Назначение            | Уровень     |
| ---------- | --------------------- | ----------- |
| `ruff`     | Линтер + форматтер    | Блокирующий |
| `mypy`     | Статическая типизация | Блокирующий |
| `pytest`   | Тесты                 | Блокирующий |
| `bandit`   | Безопасность          | Блокирующий |

Ни один из этих инструментов нельзя обойти флагом `--no-verify`
или аналогичными без явного указания в спеке.

### Требования к коду

- **Типизация:** type hints обязательны на всех публичных функциях и методах
- **Документация:** docstring обязателен на всех публичных классах и функциях
- **Логирование:** только `loguru` — `print()` запрещён категорически
- **Исключения:** нельзя глушить исключения через голый `except: pass`
- **Язык:** комментарии, docstrings, сообщения об ошибках — **русский**

## 5. Технологический стек

### Стандарт для новых модулей

| Назначение               | Библиотека                            |
| ------------------------ | ------------------------------------- |
| Модели данных            | Pydantic v2                           |
| HTTP-сервер (если нужен) | FastAPI                               |
| База данных (если нужна) | SQLAlchemy 2.0 async                  |
| Абстракции / интерфейсы  | `Protocol` (structural subtyping)     |
| CLI                      | следовать паттерну оригинального репо |

### Правило добавления зависимостей

Новая зависимость требует:
1. Явного указания в спеке (`## Dependencies`)
2. Обоснования почему нельзя обойтись существующим
3. Указания точной версии

Агент не может добавить зависимость в `pyproject.toml` или `requirements`
без выполнения всех трёх условий.

## 8. BSL-домен: правила работы с 1С кодом

### Обязательные знания перед реализацией

Агент должен понимать и соблюдать BSL-семантику:
- Типы модулей: `CommonModule`, `ObjectModule`, `ManagerModule`,
  `FormModule`, `SessionModule`, `ManagedApplicationModule`
- Видимость методов: `Экспорт` vs без экспорта
- Контекст: глобальный vs локальный
- Иерархия метаданных: Подсистема → Объект → Модуль → Метод

**Агент не имеет права:**
- Придумывать имена BSL-методов — только из существующего кода или тестов
- Хардкодить пути к 1С конфигурациям — только через переменные окружения
  или параметры конфигурации

### Поведение при ошибках парсинга

Если BSL-модуль не парсится (синтаксическая ошибка, неизвестный тип):
- Пропустить невалидный модуль
- Записать в лог: путь к файлу, тип ошибки, строку
- Продолжить обработку остальных модулей
- Включить список пропущенных модулей в итоговый результат

## 9. Безопасность и тестируемость: абсолютные правила

| Правило                        | Без исключений                             |
| ------------------------------ | ------------------------------------------ |
| Реальный LLM API в тестах      | ❌ Только моки (`pytest-mock`, `responses`) |
| BSL-фикстуры в тестах          | ❌ Только inline строки, не `.bsl` файлы    |
| API-ключи в коде или конфиге   | ❌ Только через переменные окружения        |
| Пути к 1С конфигурациям в коде | ❌ Только через параметры                   |

Нарушение любого из этих правил — блокирующая проблема,
требующая немедленного исправления до мержа.

## 10. Поведение агента при неопределённости

### Если спека неоднозначна
**Стоп.** Агент объясняет что именно непонятно и ждёт уточнения.
Агент не делает предположений и не продолжает реализацию.

### Если задача выходит за рамки спеки
**Стоп.** Агент сообщает что обнаружил scope creep и предлагает
создать новую спеку для дополнительного функционала.

### Если есть противоречие между спекой и конституцией
**Конституция побеждает.** Агент сообщает о противоречии
и не реализует то, что нарушает конституцию.

### Прогресс
После каждой выполненной задачи из `tasks.md` агент:
1. Отмечает задачу выполненной `[x]`
2. Кратко сообщает что сделано
3. Называет следующую задачу

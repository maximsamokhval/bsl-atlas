# Feature Specification: DeepSeek LLM Provider Integration

**Feature Branch**: `001-deepseek-llm-provider`  
**Created**: 2026-03-04  
**Status**: Draft  
**Input**: User description: "Реализовать поддержку DeepSeek API в качестве провайдера LLM в текущем проекте, строго соблюдая архитектурные принципы и ограничения, изложенные в constitution.md."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Использование DeepSeek API для генерации текста (Priority: P1)

Как разработчик, я хочу использовать DeepSeek API для генерации текста в рамках системы, чтобы интегрировать современные LLM‑возможности в анализ BSL‑кода и документации.

**Why this priority**: DeepSeek предоставляет мощные и экономичные модели, которые могут значительно улучшить качество генерации текста (например, для авто‑документирования, ответов на вопросы по коду). Это прямо соответствует цели проекта «Интеграция с AI‑агентами через MCP».

**Independent Test**: Можно протестировать, отправив тестовый запрос к API DeepSeek через новый провайдер и убедившись, что возвращается корректный текстовый ответ. Тест не требует других провайдеров или изменения существующего кода.

**Acceptance Scenarios**:

1. **Given** система настроена с валидным API‑ключом DeepSeek  
   **When** пользователь отправляет текстовый запрос через LLM‑провайдер  
   **Then** система возвращает сгенерированный ответ от DeepSeek API

2. **Given** API‑ключ DeepSeek отсутствует или неверен  
   **When** пользователь отправляет текстовый запрос  
   **Then** система возвращает понятную ошибку, не падая

### User Story 2 - Конфигурируемый провайдер эмбеддингов (Priority: P2)

Как оператор, я хочу выбирать провайдера эмбеддингов между Ollama и Deepinfra Qwen через параметры проекта, чтобы управлять балансом стоимости, скорости и качества векторизации.

**Why this priority**: Разные окружения (разработка, продакшен) требуют разных провайдеров эмбеддингов. Возможность выбора уменьшает эксплуатационные расходы и повышает гибкость.

**Independent Test**: Можно протестировать, изменив переменную окружения `EMBEDDING_PROVIDER` (или `INDEXING_PROVIDER`) на `deepinfra` (или `ollama`) и убедившись, что система создаёт эмбеддинги через выбранный провайдер, а все индексационные операции работают.

**Acceptance Scenarios**:

1. **Given** в конфигурации указан `EMBEDDING_PROVIDER=ollama`  
   **When** система выполняет индексацию BSL‑модулей  
   **Then** эмбеддинги создаются через локальный Ollama

2. **Given** в конфигурации указан `EMBEDDING_PROVIDER=deepinfra`  
   **When** система выполняет индексацию BSL‑модулей  
   **Then** эмбеддинги создаются через Deepinfra API (Qwen3‑Embedding‑4B)

### User Story 3 - Соблюдение ограничений проекта (Priority: P3)

Как ответственный за качество, я хочу, чтобы реализация строго соблюдала три ключевых ограничения конституции проекта (Brownfield, TDD, стандарты качества), чтобы сохранить целостность кодовой базы и гарантировать надёжность.

**Why this priority**: Нарушение любого из этих ограничений ставит под угрозу стабильность форка и увеличивает технический долг. Соблюдение гарантирует, что новая функциональность интегрируется безопасно и предсказуемо.

**Independent Test**: Можно проверить, что все новые файлы проходят линтер (`ruff`), типизацию (`mypy`), тесты (`pytest`) и безопасность (`bandit`); что каждый новый модуль сопровождается тестами, написанными до реализации (TDD); и что ни один существующий файл оригинального репозитория не изменён (только добавлены новые).

**Acceptance Scenarios**:

1. **Given** репозитория содержит оригинальный код  
   **When** реализация завершена  
   **Then** в `git diff` нет изменений в файлах, принадлежащих оригинальному репо (кроме разрешённых в спеке)

2. **Given** новая функциональность  
   **When** запускается `pytest`  
   **Then** все тесты проходят, включая тесты, написанные до реализации (red‑green‑refactor)

3. **Given** новый код  
   **When** выполняются `ruff`, `mypy`, `bandit`  
   **Then** проверки проходят без ошибок

---

### Edge Cases

- Что происходит, когда DeepSeek API возвращает ошибку (например, rate limit, недоступность)?  
  Система должна логировать ошибку и возвращать понятное сообщение пользователю, не прерывая работу других компонентов.

- Как система ведёт себя, если выбранный провайдер эмбеддингов временно недоступен?  
  Должна быть возможность использовать fallback‑провайдер (например, Ollama как резерв для Deepinfra) – если эта возможность уже предусмотрена существующей абстракцией.

- Что делать, если конфигурация содержит недопустимое значение провайдера?  
  Система должна валидировать конфигурацию при старте и сообщать об ошибке, не начиная индексацию.

- Как обрабатываются большие тексты, превышающие лимиты токенов DeepSeek?  
  Система должна разделять входные данные на чанки в соответствии с лимитами модели (используя существующие паттерны чанкинга, если они есть).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST поддерживать DeepSeek API как провайдера LLM для текстовой генерации (chat/completion).
- **FR-002**: Система MUST позволять выбирать провайдера эмбеддингов между `ollama` и `deepinfra` через конфигурационные параметры (переменные окружения или конфигурационный файл).
- **FR-003**: Реализация MUST следовать существующей абстракции провайдеров (протокол `EmbeddingProvider` и фабрика `create_embedding_provider`), расширяя её для нового провайдера LLM, если необходимо.
- **FR-004**: Реализация MUST явно учитывать три ключевых ограничения конституции:  
  (1) Brownfield – не изменять существующий код оригинального репозитория, только добавлять новый;  
  (2) TDD – тесты пишутся до реализации, каждый цикл red‑green‑refactor;  
  (3) Стандарты качества – линтер (`ruff`), типизация (`mypy`), тесты (`pytest`), безопасность (`bandit`), документация на русском.
- **FR-005**: Конфигурационные файлы и документация MUST быть обновлены для поддержки нового провайдера (примеры переменных окружения, описание параметров).
- **FR-006**: Система MUST обрабатывать ошибки API (сетевые, аутентификации, лимиты) и предоставлять ясные сообщения в логах.

### Key Entities

- **LLMProvider**: Абстракция провайдера языковых моделей (если ещё не существует, должна быть введена). Содержит метод `generate(prompt: str, **kwargs) → str`.
- **DeepSeekProvider**: Конкретная реализация LLMProvider для DeepSeek API. Использует официальный REST‑эндпоинт, поддерживает необходимые параметры (temperature, max_tokens и т.д.).
- **EmbeddingProvider**: Существующий протокол для создания эмбеддингов. Должен быть расширен для поддержки `deepinfra` как нового значения провайдера.
- **Configuration**: Существующий класс конфигурации (`Config`). Должен быть расширен для поддержки новых переменных окружения (`DEEPSEEK_API_KEY`, `DEEPSEEK_BASE_URL`, `EMBEDDING_PROVIDER=deepinfra`).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Пользователь может отправить текстовый запрос через DeepSeek API и получить корректный ответ – успешный HTTP‑запрос, парсинг ответа, возврат текста (измеряется прохождением интеграционного теста).
- **SC-002**: Оператор может изменить провайдера эмбеддингов с `ollama` на `deepinfra` (или обратно) только через переменные окружения, без изменения кода – после изменения конфигурации система создаёт эмбеддинги через выбранный провайдер (измеряется прохождением конфигурационного теста).
- **SC-003**: Все новые модули проходят обязательные проверки: `ruff` (без ошибок), `mypy` (типы соответствуют), `pytest` (все тесты зелёные), `bandit` (без security issues) – измеряется автоматическим пайплайном CI.
- **SC-004**: Реализация не нарушает существующую функциональность – все существующие тесты (`pytest`) продолжают проходить после добавления нового кода (регрессия отсутствует).
- **SC-005**: Документация обновлена – в `README.md` или отдельном конфигурационном руководстве описаны шаги по настройке DeepSeek и Deepinfra провайдеров.

**Date**: 2026-03-04  
**Feature**: 001-deepseek-llm-provider  
**Purpose**: Устранить неопределённости, выявленные в Technical Context, и зафиксировать ключевые технологические решения.

## Неопределённости (Unknowns)

В Technical Context отсутствуют маркеры `NEEDS CLARIFICATION`. Все технологические параметры определены:

1. **Язык и версия**: Python 3.11+ (подтверждено `pyproject.toml`).
2. **Основные зависимости**: `httpx` для HTTP‑запросов к DeepSeek API, `pydantic` для валидации, `loguru` для логирования, `chromadb` для векторного хранилища, `openai` (клиент) для совместимости с существующей абстракцией, `ollama` для локальных эмбеддингов.
3. **Хранилище**: SQLite (через SQLAlchemy 2.0 async) и ChromaDB – уже используются в проекте.
4. **Тестирование**: `pytest` с мокированием (`pytest‑mock`, `responses`) – соответствует конституции (реальные API в тестах запрещены).
5. **Целевая платформа**: Linux/macOS, Docker – соответствует текущей инфраструктуре.
6. **Тип проекта**: библиотека/CLI‑утилита с MCP‑сервером – соответствует назначению bsl‑atlas.
7. **Цели производительности**: таймаут 30 сек на запрос к DeepSeek API, индексация эмбеддингов без деградации – разумные допущения.
8. **Ограничения**: Brownfield, TDD, стандарты качества – уже проверены в Constitution Check.

## Исследование API DeepSeek

### Решение 1: Эндпоинт и аутентификация

- **Decision**: Использовать официальный REST API DeepSeek (`https://api.deepseek.com/v1/chat/completions`).
- **Rationale**: API совместимо с OpenAI‑форматом, что упрощает интеграцию через существующий клиент `openai` (или прямой HTTP‑запрос с `httpx`). Аутентификация через заголовок `Authorization: Bearer <API_KEY>`.
- **Alternatives considered**:
  - Использование SDK DeepSeek (не представлен) – не рекомендуется из‑за излишней зависимости.
  - Прокси через сервисы типа OpenRouter – добавит лишнее звено и стоимость.

### Решение 2: Модель эмбеддингов Deepinfra

- **Decision**: Использовать эндпоинт `https://api.deepinfra.com/v1/inference/Qwen/Qwen3-Embedding-4B` для создания эмбеддингов.
- **Rationale**: Модель Qwen3‑Embedding‑4B показывает высокое качество для многоязычных текстов (включая русский), бесплатна для ограниченного объема. API также совместимо с OpenAI‑форматом.
- **Alternatives considered**:
  - Оставить только Ollama – уменьшит гибкость и потребует локальных ресурсов.
  - Использовать другие провайдеры (Cohere, Jina) – потребует дополнительных интеграций без явной выгоды.

### Решение 3: Абстракция LLMProvider

- **Decision**: Ввести новый протокол `LLMProvider` с методом `generate(prompt: str, **kwargs) -> str`, аналогичный существующему `EmbeddingProvider`.
- **Rationale**: Единообразие с архитектурой провайдеров проекта, упрощает добавление новых LLM‑провайдеров в будущем.
- **Alternatives considered**:
  - Прямое использование клиента `openai` внутри бизнес‑логики – нарушит принцип инверсии зависимостей и усложнит тестирование.
  - Использование наследования от базового класса – протоколы (structural subtyping) более гибки и соответствуют конституции (использование `Protocol`).

### Решение 4: Конфигурационные переменные

- **Decision**: Добавить переменные окружения:
  - `DEEPSEEK_API_KEY` (обязательный)
  - `DEEPSEEK_BASE_URL` (опциональный, по умолчанию `https://api.deepseek.com`)
  - `EMBEDDING_PROVIDER` (расширить допустимые значения: `ollama`, `deepinfra`)
- **Rationale**: Следует существующему паттерну конфигурации (`config.py`). Позволяет менять провайдеров без изменения кода.
- **Alternatives considered**:
  - Хардкодить URL и ключи в коде – нарушает конституцию (запрет хардкодинга секретов).
  - Использовать отдельный конфигурационный файл – излишне усложняет развертывание.

### Решение 5: Обработка ошибок и отказоустойчивость

- **Decision**: Использовать существующий механизм `FallbackEmbeddings` для эмбеддингов; для LLM‑провайдера реализовать повторные запросы (retry) с экспоненциальной задержкой и ясным логированием через `loguru`.
- **Rationale**: Соответствует принципу надёжности и существующим паттернам проекта.
- **Alternatives considered**:
  - Игнорировать сетевые ошибки – приведёт к нестабильности.
  - Прерывать всю операцию при первой ошибке – ухудшит пользовательский опыт.
